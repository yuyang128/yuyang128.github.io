How HTTP works: HTTP 的工作原理：

All HTTP transactions follow the same general format. Each client request and server response has three parts: the request or response line, a header section and the entity body.所有 HTTP 事务遵循相同的通用格式。每个客户端请求和服务器响应包含三个部分：请求或响应行、头部部分和实体体。
The client initiates a transaction as follows:客户端发起交易的方式如下：
  • The client contacts the server and sends a document request. A GET request can have url parameters and those parameters will be available in the web access logs.客户端联系服务器并发送文档请求。GET 请求可以有 URL 参数，这些参数会在网页访问日志中提供。
    • GET /index.html?param=value HTTP/1.0GET /index.html？param=value HTTP/1.0
  • Next, the client sends optional header information to inform the server of its configuration and the document formats it will accept.接着，客户端发送可选的头部信息，以告知服务器其配置及其可接受的文档格式。
    • User-Agent: Mozilla/4.06 Accept: image/gif,image/jpeg,用户代理：Mozilla/4.06 接受：image/gif，image/jpeg，/
  • In a POST request, the user supplied data will follow the optional headers and is not part of the contained within the POST URL.在 POST 请求中，用户提供的数据将跟随可选的头部，而不是 POST URL 中包含的内容。

The CIA Triad

这一章都是理论，后面测验也很简单。
 
The CIA Triad (confidentiality, integrity, availability) is a model for information security. The three elements of the triad are considered the most crucial information security components and should guarantee in any secure system.CIA 三位一体（机密性、完整性、可用性）是信息安全的模型。三位一体中的三个要素被认为是最关键的信息安全组成部分，任何安全系统都应当得到保障。Serious consequences can result if even one of these elements is breached.如果其中哪怕一个被违反，都可能带来严重后果。
The CIA Triad was created to provide a baseline standard for evaluating and implementing security regardless of the underlying system and/or organization.CIA 三位一会旨在为评估和实施安全提供基线标准，无论底层系统和/或组织为何。
 

Confidentiality 保密

Confidentiality is "the property that information is not made available or disclosed to unauthorized individuals, entities, or processes." In other words, confidentiality requires that unauthorized users should not be able to access sensitive resources. Confidentiality must be balanced with availability; authorized persons must still access the resources they have been granted permissions for.保密性是指“信息不被未经授权的个人、实体或流程公开或披露的属性”。换句话说，保密性要求未经授权的用户不得访问敏感资源。保密性必须与可用性取得平衡;授权人员仍需访问其获授权的资源。
Although confidentiality is similar to "privacy," these two words are not interchangeable. Instead, confidentiality is a component of privacy; confidentiality is implemented to protect resources from unauthorized entities.虽然保密与“隐私”类似，但这两个词不能互换。相反，保密性是隐私的一个组成部分;保密性是为了保护资源免受未经授权的机构侵害。

Examples that compromise confidentiality:危害保密性的例子：

  • a hacker gets access to the password database of a company黑客入侵了一家公司的密码数据库
  • a sensitive email is sent to the incorrect individual一封敏感邮件发送给了错误的人
  • a hacker reads sensitive information by intercepting and eavesdropping on an information transfer黑客通过拦截和窃听信息传输来读取敏感信息

Examples of methods ensuring confidentiality确保保密的方法示例

  • data encryption 数据加密
  • properly implemented authentication and access control正确实现的认证和访问控制
    • securely stored passwords安全存储的密码
    • multi-factor authentication (MFA)多因素认证（MFA）
    • biometric verification 生物识别验证
  • minimizing the number of places/times the information appears尽量减少信息出现的地点和时间
  • physical security controls such as properly secured server rooms物理安全控制，如妥善安全的服务器机房
 

Integrity 完整性

Integrity is "the property of accuracy and completeness." In other words, integrity means maintaining the consistency, accuracy, and trustworthiness of data over its entire life cycle. Data must not change during transit, and unauthorized entities should not alter the data.完整性是“准确性和完整性的属性”。换句话说，完整性意味着在整个生命周期内保持数据的一致性、准确性和可信度。数据在传输过程中不得发生变化，未经授权的实体也不得更改数据。

Examples that compromise integrity:影响完整性的例子：

  • human error when entering data录入数据时的人为错误
  • errors during data transmission数据传输过程中的错误
  • software bugs and hardware failures软件漏洞和硬件故障
  • hackers change information that they should not have access to黑客篡改了他们本不该接触的信息

Examples of methods ensuring the integrity确保完整性的方法示例

  • well functioning authentication methods and access control运行良好的认证方法与访问控制
  • checking integrity with hash functions用哈希函数检查完整性
  • backups and redundancy 备份与冗余
  • auditing and logging 审计与记录
 

Availability 可获得性

Availability is "the property of being accessible and usable on demand by an authorized entity." In other words, authorized persons should have access to permitted resources at all times.可用性是“授权实体按需可访问和可用的属性”。换句话说，授权人员应随时获得允许的资源。

Examples that compromise availability:影响可用性的例子：

  • denial-of-service attacks (DOS)拒绝服务攻击（DOS）
  • hardware failures 硬件故障
  • fire or other natural disasters火灾或其他自然灾害
  • software or network misconfigurations软件或网络配置错误

Examples of methods ensuring availability确保可用性的方法示例

  • intrusion detection systems (IDSs)入侵检测系统（IDS）
  • network traffic control 网络交通控制
  • firewalls 防火墙
  • physical security of hardware and underlying infrastructure硬件及底层基础设施的物理安全
    • protection against fire, water, and other elements防火、防水及其他因素的防护
  • hardware maintenance 硬件维护
  • redundancy 冗余

Using Indrect References使用 Indrect 引用

Not many applications employ it, but you can use indirect references. In this case you can run your references across a hashing, encoding or other function on the server so that the id that the client sees is not the actual reference which the server handles. This will reduce efficiency some (a common trade-off for security) and is still subject to being guessed, brute-forced or reverse engineered.
虽然很少应用，但你可以使用间接引用。在这种情况下，你可以在服务器端通过哈希、编码或其他函数运行引用，使客户端看到的 ID 不是服务器实际处理的引用。这会降低效率（这是安全性的常见权衡），但仍可能被猜测、暴力破解或逆向工程。
This approach should not be the only protection used. It can be used as an additional layer. Your server must implement the logic of mapping client (indirect) to server (direct) references.
这种方法不应是唯一的保护措施。它可以作为额外的层。你的服务器必须实现将客户端（间接）映射到服务器（直接）引用的逻辑。

Missing Function Level Access Control

Access control, like preventing XSS with output encoding, can be tricky to maintain. One must ensure it is adequately enforced throughout the entire application, thus in every method/function.
访问控制，比如用输出编码防止 XSS 的，维护起来可能很棘手。必须确保该机制在整个应用中得到充分执行，因此在每个方法/函数中都必须如此。

IDOR vs Missing Function Level Access ControlIDOR 与缺失功能级访问控制

The fact is many people (including the author of this lesson) would combine function level access control and IDOR into 'Access Control.' For the sake of OWASP Top 10 and these lessons, we will make a distinction. The distinction most made is that IDOR is more of a 'horizontal' or 'lateral' access control issue, and missing function level access control 'exposes functionality.' Even though the IDOR lesson here demonstrates how functionality may also be exposed (at least to another user in the same role), we will look at other ways functionality might be exposed.事实上，许多人（包括本课程作者）会将功能级访问控制和 IDOR 合并为“访问控制”。为了 OWASP 十大和这些课程的理解，我们将做出区分。最常见的区分是，IDOR 更偏向“水平”或“横向”访问控制问题，而缺失功能级访问控制则“暴露功能”。尽管这里的 IDOR 课程展示了功能可能被暴露（至少对同一角色的其他用户），我们还会探讨其他功能可能被暴露的方式。

当你请求 text/html 时：Spring 会尝试寻找一个名为 list_users.html 的模板文件，将用户列表塞进去渲染成网页。但在这一关，这个文件被删除了，所以报了 500 错误。 当你请求 application/json 时：Spring 会跳过模板引擎，直接使用 Jackson 库 把 Java 的 List<User> 对象转换成纯文本的 JSON 字符串发送给你。 为什么不能是别的类型？ 如果你尝试其他的 Content-Type 或 Accept，结果可能会完全不同：请求类型 (Accept)后端反应结果text/html寻找 HTML 模板进行渲染失败 (500)，因为模板文件缺失。application/xml寻找 XML 转换器可能成功，也可能报 406 Not Acceptable（如果没装 XML 库）。text/plain尝试转换成纯文本通常会失败，因为复杂对象（用户列表）没有默认的文本转换逻辑。application/json直接序列化对象成功！ 这是现代 API 的标准语言，几乎所有后端都默认开启。
这其实是一种逻辑降级”。当我们发现发现的网页界面（View）坏了，就直接去请求底层的“数据接口”（Model）。
很多开发者只给 HTML 页面做了权限检查，却忘记了同一个 URL 的 JSON 返回路径也需要权限保护。你刚才的操作实际上是：
  1. 绕过了视图层 (View Layer)。
  1. 直接探测到了模型层 (Model Layer)
 
那么当你把accept改成了application/json，发现还有问题。
这个时候是因为没有指定Content-Type 。
为什么一定要 Content-Type？
虽然 GET 请求通常不带数据（Body），不需要 Content-Type，但在 WebGoat 这个特定的环境下：
  • 加上 Content-Type: application/json 是为了彻底消除歧义。
    • 它强制 Spring 的 MessageConverter 进入“JSON 模式”，确保不会因为某些默认配置而再次跑去寻找那个不存在的 HTML 文件。
